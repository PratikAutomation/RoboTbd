3. How RoboLink Normalizes Different Vendors

Each robot vendor uses different data formats, units, and naming conventions. RoboLink translates everything into one common schema automatically.

UR5 Raw Data

Different format

obs.state.manipulator.qpos[6] obs.state.manipulator.qtorque[6] obs.state.manipulator.qvel[6] Units: radians, Nm, rad/s
→

RoboLink Device Profile

profiles/universal_robots.yaml

node_mapping: joints: position: "qpos[{i}]" torque: "qtorque[{i}]" velocity: "qvel[{i}]" unit_conversions: position: null # already rad
→

Common Schema

Identical format

joint_1: position_rad: 0.0000 velocity_rad_s: 0.0000 torque_nm: 0.0000 temperature_c: null error_code: 0

SO-ARM100 Raw Data

Completely different format

observation.state[6] Units: servo ticks (raw encoder) No torque data available No velocity data available
→

RoboLink Device Profile

profiles/so_arm100.yaml

node_mapping: joints: position: "state[{i}]" unit_conversions: position: from: "servo_ticks" to: "radians" factor: 0.0174533
→

Common Schema

Same format as UR5!

joint_1: position_rad: -0.3037 velocity_rad_s: null torque_nm: null temperature_c: null error_code: 0

Two vendors. Two data formats. One common schema. One dashboard.
